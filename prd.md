# PRD.md

## 1. 제품 개요

### 1.1 제품명
진영쌤이 만든 웹앱모음

### 1.2 제품 설명
학교에서 필요한 여러 웹앱을 한 곳에 모아 보여주고, 사용자가 카테고리별로 탐색한 뒤 각 앱을 바로 실행할 수 있는 웹 포털을 구축한다.

이 포털은 단순한 링크 모음이 아니라, 제작 목적별로 앱을 정리하고 빠르게 실행할 수 있는 런치형 포털을 목표로 한다. 또한 운영자는 별도 개발 작업 없이 관리자 코드 인증 후 앱 정보를 직접 등록할 수 있어야 한다.

### 1.3 목표
- 사용자가 전체 앱을 한눈에 보고 빠르게 실행할 수 있어야 한다.
- 카테고리별 탐색이 쉬워야 한다.
- 운영자는 관리자 인증 후 앱을 직접 등록할 수 있어야 한다.
- 새 앱 등록 시 포털에 빠르게 반영되어야 한다.

### 1.4 비목표
초기 버전에서 제외하는 항목은 다음과 같다.
- 사용자 로그인/회원가입
- 사용자 리뷰/평점
- 개인화 추천
- 결제/구독 기능
- 커뮤니티
- 고급 통계 대시보드
- 복잡한 권한 체계(다중 관리자 권한 레벨)

## 2. 문제 정의

학습용 웹앱이 늘어날수록 다음 문제가 발생한다.
- 어떤 앱이 있는지 한눈에 보기 어렵다.
- 앱 링크가 분산되어 있어 접근성이 떨어진다.
- 비슷한 성격의 앱끼리 묶어서 보기 어렵다.
- 새 앱을 추가할 때 매번 개발자가 직접 수정해야 하면 운영이 비효율적이다.

이 포털은 다음 방식으로 문제를 해결한다.
- 전체 앱 목록 제공
- 카테고리 기반 탐색 제공
- 앱 카드에서 바로 실행 가능
- 관리자 코드 기반의 간단한 앱 등록 기능 제공

## 3. 대상 사용자

### 3.1 일반 사용자
- 학습용 웹앱을 사용하는 사용자
- 원하는 앱을 빠르게 찾고 바로 실행하고 싶은 사용자

### 3.2 운영자(관리자)
- 포털에 새 앱을 추가/수정하고 싶은 관리자
- 관리자 코드 인증 후 직접 등록을 수행하는 사용자

---

## 4. 핵심 사용자 시나리오

### 시나리오 1: 전체 앱 보기
1. 사용자가 포털 메인 또는 전체 앱 페이지에 접속한다.
2. 등록된 앱 목록을 카드 형태로 확인한다.
3. 원하는 앱을 찾는다.
4. 실행 버튼을 눌러 앱으로 이동한다.

### 시나리오 2: 카테고리로 앱 찾기
1. 사용자가 카테고리를 선택한다.
2. 선택한 카테고리에 해당하는 앱만 표시된다.
3. 앱 카드를 보고 원하는 앱을 실행한다.

### 시나리오 3: 관리자가 앱 추가하기
1. 운영자가 관리자 진입 버튼을 누른다.
2. 관리자 코드 입력 화면이 나타난다.
3. 올바른 코드 입력 시 앱 등록 폼으로 이동한다.
4. 제목, 카테고리, 링크, 썸네일을 입력한다.
5. 저장 버튼을 누른다.
6. 앱이 목록에 반영된다.

## 5. 핵심 기능

### 5.1 전체 앱 목록
- 등록된 모든 앱을 카드 형태로 표시한다.
- 카드에는 최소 다음 정보가 포함된다.
  - 썸네일
  - 제목
  - 카테고리
  - 실행 버튼
- 필요 시 짧은 설명도 포함할 수 있다.

### 5.2 카테고리 탐색
- 사용자는 카테고리별로 앱을 필터링할 수 있다.
- 카테고리는 상단 탭 또는 좌측 사이드바로 제공한다.
- 카테고리 예시:
  - 식생활 
  - 의생활
  - 가족
  - 소비
  - 주거
  - 청소년
  - 학급운영
  - 업무관련 
  - 기타


### 5.3 앱 런치 기능
- 각 앱 카드에는 실행 버튼이 있어야 한다.
- 클릭 시 앱 링크로 이동한다.
- 외부 링크는 새 탭 열기를 기본값으로 고려한다.

### 5.4 관리자 코드 인증
- 관리자는 관리자 전용 진입 화면에서 코드를 입력한다.
- 코드가 일치하면 앱 등록 기능에 접근할 수 있다.
- 코드가 일치하지 않으면 접근이 차단된다.
- 초기 버전은 단일 공통 관리자 코드 방식으로 구현한다.

### 5.5 관리자 앱 등록 기능
관리자는 인증 후 앱 등록 폼에서 다음 항목을 입력할 수 있어야 한다.
- 제목
- 카테고리 선택
- 앱 링크(URL)
- 썸네일 이미지 업로드

저장 시 해야 하는 동작:
- 입력값 검증
- 썸네일 저장
- 앱 데이터 저장
- 전체 앱 목록에 반영

### 5.6 관리자 앱 수정/삭제 기능
초기 버전에서는 최소한 다음 중 하나를 지원하는 것이 좋다.
- 등록된 앱 비활성화
- 등록된 앱 수정
- 등록된 앱 삭제

MVP에서는 우선순위를 다음과 같이 둔다.
1. 앱 추가
2. 앱 수정
3. 앱 비활성화 또는 삭제

## 6. 관리자 등록 플로우

### 6.1 진입
- 헤더 또는 별도 URL에서 관리자 페이지 진입
- 예시 경로: `/admin`

### 6.2 인증
- 관리자 코드 입력 필드 제공
- 확인 버튼 클릭 시 코드 검증
- 성공 시 등록 폼 표시
- 실패 시 에러 메시지 표시

### 6.3 등록 폼
필드 구성:
- 앱 제목: 텍스트 입력
- 카테고리: 드롭다운 선택
- 앱 링크: URL 입력
- 썸네일: 파일 업로드

선택 필드:
- 짧은 설명
- 대표 앱 여부
- 노출 여부

### 6.4 저장 후 반영
- 저장 성공 시 “앱이 등록되었습니다” 메시지 노출
- 전체 앱 목록에 새 앱이 반영됨
- 필요 시 캐시 갱신 또는 새로고침 유도

## 7. 기능 요구사항

### 7.1 일반 사용자 기능
1. 전체 앱 목록을 볼 수 있어야 한다.
2. 카테고리별로 앱을 필터링할 수 있어야 한다.
3. 앱 카드를 눌러 앱을 실행할 수 있어야 한다.

### 7.2 관리자 기능
1. 관리자는 관리자 코드로 인증할 수 있어야 한다.
2. 관리자는 앱 제목을 입력할 수 있어야 한다.
3. 관리자는 카테고리를 선택할 수 있어야 한다.
4. 관리자는 앱 링크를 입력할 수 있어야 한다.
5. 관리자는 썸네일 이미지를 업로드할 수 있어야 한다.
6. 저장 후 앱이 포털에 반영되어야 한다.

## 8. 입력값 검증 규칙

### 8.1 제목
- 필수 입력
- 최소 1자 이상
- 최대 길이는 추후 정의(예: 50자)

### 8.2 카테고리
- 필수 선택
- 사전 정의된 목록 중 하나 선택

### 8.3 링크(URL)
- 필수 입력
- 올바른 URL 형식이어야 함
- `http://` 또는 `https://` 포함

### 8.4 썸네일
- 필수 여부는 정책에 따라 결정
- 허용 확장자: jpg, jpeg, png, webp
- 최대 용량 제한 필요(예: 2MB 또는 5MB)

### 8.5 관리자 코드
- 필수 입력
- 서버 측 검증 필요
- 프론트엔드에 평문으로 노출되면 안 됨

## 13. 비기능 요구사항

### 13.1 보안
- 관리자 코드는 서버 측에서 검증해야 한다.
- 업로드 파일 형식 검증이 필요하다.
- 악성 파일 업로드 방지 필요
- 관리자 기능 URL은 직접 접근 시에도 인증 검증이 필요하다.

### 13.2 성능
- 앱 목록 로딩은 빠르게 이루어져야 한다.
- 썸네일 최적화 필요
- 저장 후 반영까지의 시간이 짧아야 한다.

### 13.3 반응형
- 모바일에서도 앱 목록 탐색 가능해야 한다.
- 관리자 등록 폼도 모바일에서 사용 가능해야 한다.

### 13.4 확장성
- 카테고리 추가가 쉬워야 한다.
- 향후 설명, 태그, 정렬 우선순위 같은 필드 추가가 가능해야 한다.

## 17. 개인적 제안

다음은 의견이며, 확정 사양은 아니다.
- 초기에는 관리자 코드 1개 + 앱 추가 기능만 먼저 구현하는 것이 가장 빠르고 안정적이다.
- 수정/삭제, 다중 관리자, 승인 프로세스는 2차 버전으로 미루는 것이 적절하다.
- 메인 페이지는 과하게 복잡하게 만들기보다 전체 앱과 카테고리 진입을 명확하게 보여주는 방향이 좋다.
