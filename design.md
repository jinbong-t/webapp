# design.md

## 1. 문서 목적

이 문서는 학습 웹앱 포털의 UI/UX 설계 원칙, 화면 구조, 컴포넌트 규칙, 시각 스타일 방향을 정의한다. 본 문서의 목적은 다음과 같다.

- 제품의 화면 구조를 일관되게 설계한다.
- 전체 앱, 카테고리, 앱 실행 중심의 UX를 명확히 한다.
- 관리자 등록 플로우까지 포함한 UI 기준을 정리한다.
- 개발자와 디자이너가 동일한 기준으로 구현할 수 있게 한다.

---

## 2. 디자인 목표

### 2.1 핵심 목표
- 사용자가 앱을 빠르게 찾을 수 있어야 한다.
- 사용자가 앱을 바로 실행할 수 있어야 한다.
- 앱 수가 늘어나도 복잡해 보이지 않아야 한다.
- 관리자가 앱을 쉽게 등록할 수 있어야 한다.

### 2.2 UX 키워드
- 단순함
- 빠른 탐색
- 명확한 구조
- 일관된 카드 UI
- 관리하기 쉬운 시스템

---

## 3. 디자인 원칙

### 3.1 탐색 우선
포털의 가장 중요한 목적은 앱을 찾는 것이다. 따라서 첫 화면부터 앱 탐색과 카테고리 이동이 명확해야 한다.

### 3.2 실행 우선
각 앱은 설명보다 실행 버튼이 더 중요하다. 앱 카드는 정보 확인보다 바로 열기 행동을 유도해야 한다.

### 3.3 일관성 우선
앱 개수가 많아질수록 UI 일관성이 중요해진다. 카드 구조, 버튼 위치, 텍스트 길이, 썸네일 비율은 가능한 고정한다.

### 3.4 관리 효율성 고려
관리자가 직접 앱을 추가하는 구조이므로 썸네일, 제목, 카테고리만으로도 보기 좋은 레이아웃이 유지되어야 한다.

### 3.5 모바일 우선 대응
사용자는 모바일에서도 앱을 탐색할 수 있어야 하므로, 카테고리와 앱 카드 구조는 작은 화면에서도 유지 가능해야 한다.

## 4. 전체 경험 구조

사용자 경험은 아래 흐름을 기본으로 한다.

1. 포털에 접속한다.
2. 전체 앱 또는 카테고리로 탐색한다.
3. 앱 카드를 본다.
4. 실행 버튼을 누른다.
5. 앱으로 이동한다.

관리자 경험은 아래 흐름을 기본으로 한다.

1. 관리자 페이지 진입
2. 관리자 코드 입력
3. 등록 폼 작성
4. 저장
5. 목록 반영 확인

## 6. 화면 구조 제안

### 6.1 홈 화면

#### 목적
- 포털의 성격을 바로 이해시키기
- 카테고리 탐색 시작점 제공
- 대표 앱 또는 최신 앱 노출
- 전체 앱 페이지로 자연스럽게 연결

#### 구성 요소
- 헤더
- 메인 타이틀
- 검색창
- 카테고리 퀵탭
- 대표 앱 섹션
- 전체 앱 미리보기
- 전체 앱 보기 버튼
- 푸터

#### UX 포인트
- 첫 화면에서 스크롤을 많이 하지 않아도 주요 앱을 볼 수 있어야 한다.
- 사용자가 여기서 무엇을 할 수 있는지 3초 안에 이해할 수 있어야 한다.

### 6.2 전체 앱 페이지

#### 목적
- 등록된 앱을 한눈에 보여주기
- 카테고리와 검색으로 빠르게 필터링하기
- 가장 자주 사용하는 메인 탐색 페이지 역할 수행

#### 구성 요소
- 상단 헤더
- 페이지 제목
- 검색창
- 카테고리 필터
- 앱 카드 그리드

#### 레이아웃 제안
##### 데스크톱
- 좌측: 카테고리 필터
- 우측: 앱 카드 목록

##### 모바일
- 상단: 검색창
- 그 아래: 가로 스크롤 카테고리 탭
- 하단: 1열 또는 2열 카드 목록

### 6.3 카테고리 페이지

#### 목적
- 특정 학습 분야의 앱을 집중적으로 보여주기

#### 구성 요소
- 카테고리 제목
- 해당 카테고리 설명(선택)
- 앱 카드 목록

#### UX 포인트
- 사용자는 현재 어떤 카테고리를 보고 있는지 명확히 알아야 한다.
- 전체 앱으로 돌아가는 이동 경로가 명확해야 한다.

### 6.4 관리자 페이지

#### 목적
- 관리자가 앱을 쉽게 등록할 수 있게 하기

#### 단계 1: 코드 인증 화면
구성 요소:
- 페이지 제목
- 관리자 코드 입력 필드
- 확인 버튼
- 에러 메시지

#### 단계 2: 앱 등록 화면
구성 요소:
- 제목 입력
- 카테고리 선택
- 링크 입력
- 썸네일 업로드
- 저장 버튼

#### UX 포인트
- 폼은 짧고 단순해야 한다.
- 입력 실수를 줄이기 위해 필수값과 오류 메시지가 명확해야 한다.
- 저장 후 성공 여부가 즉시 보여야 한다.

## 7. 핵심 컴포넌트 설계

### 7.1 헤더
#### 역할
- 브랜드 인지
- 주요 페이지 이동
- 관리자 진입

#### 포함 요소
- 로고
- 전체 앱
- 카테고리
- 관리자

#### 원칙
- 헤더는 항상 단순해야 한다.
- 메뉴 수는 최소화한다.

### 7.2 검색창
#### 역할
- 앱 이름 또는 키워드 검색

#### 원칙
- 화면 상단에 배치한다.
- 플레이스홀더는 명확하게 작성한다.
  - 예: 앱 이름 또는 키워드를 검색하세요

### 7.3 카테고리 필터
#### 역할
- 사용자가 앱 목록을 빠르게 좁힐 수 있도록 함

#### UI 형태
- 데스크톱: 사이드바
- 모바일: 가로 스크롤 탭

#### 상태
- 기본
- hover
- active
- disabled

### 7.4 앱 카드
#### 역할
- 앱의 핵심 정보를 요약해서 보여주고 실행으로 연결

#### 필수 요소
- 썸네일
- 앱 제목
- 카테고리 배지
- 실행 버튼

#### 선택 요소
- 짧은 설명
- 새 앱 태그
- 추천 태그

#### 카드 구조 예시
1. 썸네일
2. 제목
3. 짧은 설명
4. 카테고리 배지
5. 실행 버튼

#### 원칙
- 제목은 1줄 또는 2줄 이내로 제한한다.
- 설명은 짧게 유지한다.
- 실행 버튼 위치는 모든 카드에서 동일해야 한다.

### 7.5 실행 버튼
#### 역할
- 앱으로 이동시키는 가장 중요한 액션

#### 버튼 문구 예시
- 실행하기
- 시작하기
- 열기

#### 원칙
- 시각적으로 가장 눈에 띄어야 한다.
- 1차 액션 버튼 스타일을 사용한다.

### 7.6 관리자 입력 폼
#### 필드
- 제목
- 카테고리
- 링크
- 썸네일

#### 원칙
- 한 줄에 너무 많은 정보를 넣지 않는다.
- 모바일에서 세로 흐름으로 자연스럽게 입력 가능해야 한다.
- 파일 업로드 상태를 명확히 보여준다.

## 8. 레이아웃 시스템

### 8.1 그리드
- 데스크톱: 12컬럼 기반 권장
- 태블릿: 6~8컬럼
- 모바일: 4컬럼 또는 단일 흐름

### 8.2 앱 카드 그리드
- 데스크톱: 3~4열
- 태블릿: 2~3열
- 모바일: 1~2열

### 8.3 여백
- 섹션 간 여백은 충분히 확보한다.
- 카드 내부 여백은 일관되게 유지한다.

## 9. 시각 스타일 가이드

### 9.1 전체 스타일 방향
- 깔끔함
- 학습 서비스다운 신뢰감
- 너무 장식적이지 않은 구조
- 콘텐츠 중심 UI

### 9.2 컬러 방향
#### 기본 컬러
- 메인: 블루 계열
- 서브: 그레이 계열
- 포인트: 카테고리/상태 구분용 보조 컬러

#### 추천 느낌
- Primary: 신뢰감 있는 파랑
- Background: 밝은 중성색
- Surface: 흰색 카드
- Text: 진한 회색 또는 거의 검정

### 9.3 타이포그래피
- 제목은 명확하고 단단한 인상
- 본문은 가독성 우선
- 한글 가독성이 좋은 폰트 사용 권장

예:
- Pretendard
- Noto Sans KR
- SUIT

### 9.4 아이콘
- 단순한 라인 아이콘 사용 권장
- 카테고리마다 과도한 아이콘 사용은 지양

## 10. 상태 설계

### 10.1 빈 상태
앱이 없거나 검색 결과가 없을 때는 다음을 보여준다.
- 안내 문구
- 전체 앱으로 돌아가기 버튼
- 관리자에게는 앱 추가 유도 문구

예:
- 검색 결과가 없습니다.
- 다른 키워드로 검색하거나 카테고리를 변경해보세요.

### 10.2 로딩 상태
- 앱 목록 로딩 중 스켈레톤 카드 표시
- 관리자 저장 중 버튼 비활성화 + 로딩 표시

### 10.3 오류 상태
- 링크 저장 실패
- 업로드 실패
- 관리자 코드 오류
- 네트워크 오류

오류 메시지는 구체적이어야 한다.
예:
- 관리자 코드가 올바르지 않습니다.
- 썸네일 업로드에 실패했습니다.
- URL 형식을 확인해주세요.

## 11. 반응형 가이드

### 모바일
- 검색창 최상단 배치
- 카테고리 탭은 가로 스크롤
- 앱 카드는 1열 또는 2열
- CTA 버튼은 충분히 크게 배치

### 태블릿
- 카드 수를 늘리되 가독성 유지
- 사이드바 대신 축소형 필터 가능

### 데스크톱
- 좌측 필터 + 우측 카드 목록 구조 권장
- 관리자 페이지도 한눈에 들어오게 넓게 사용 가능

---

## 12. 접근성 가이드

- 버튼과 링크는 명확하게 구분되어야 한다.
- 색상만으로 상태를 구분하지 않는다.
- 키보드로 주요 기능 탐색이 가능해야 한다.
- 입력 필드는 label을 가져야 한다.
- 오류 메시지는 텍스트로 제공해야 한다.
- 썸네일 이미지에는 대체 텍스트가 필요하다.

---

## 13. 인터랙션 가이드

### hover
- 카드 전체가 살짝 강조되도록 처리
- 버튼은 hover 시 명도 변화

### active
- 현재 선택된 카테고리는 명확히 구분

### focus
- 키보드 탐색 사용자를 위해 선명한 포커스 스타일 제공

### transition
- 과도한 애니메이션은 지양
- 빠르고 짧은 전환만 사용

## 16. 제안 사항

다음은 추천 의견이다.
- 홈은 너무 복잡하게 만들지 않는 것이 좋다.
- 포털 특성상 예쁜 UI보다 빨리 찾고 바로 실행하는 구조가 더 중요하다.
- 앱 카드는 통일감이 가장 중요하다.
- 관리자가 썸네일 크기를 제각각 올려도 레이아웃이 무너지지 않게 설계해야 한다.
- 초기에는 다크 모드보다 기본 라이트 모드 완성도를 우선하는 것이 효율적이다.
